# Site Style Research Report — Curated Chaos Scroll-Driven 3D Site

Merged research findings for the Curated Chaos / Fantastical-elements visual style change (scroll-driven 3D storytelling, typography choreography, codex-legal spectacle). Task context: see [PLAYBOOK.md](PLAYBOOK.md) plus the brand law in `PRODUCT.md` / `DESIGN.md` at repo root. Five research lenses were run independently; this report merges them.

**How to read the tags**

- **VALIDATED** — the practice was independently reported by 2+ separate organizations/authors (first-party vendor docs, distinct studios/practitioners, standards bodies, or research firms). Codrops-published tutorials by *different* authors/studios count as independent practitioner sources; a single article, or an award-record page that merely confirms an award, does not.
- **PROVISIONAL** — single-source (even if that source is first-party and authoritative, e.g. a vendor's own docs or a tool schema). Treat as implementable but verify during build.

Confidence levels (high/medium) from the original research are preserved where they qualify a rule.

## Lens 1 — Scroll-Driven 3D Engineering (GSAP ScrollTrigger + Lenis + Three.js/R3F)

2025–2026 engineering best practice for scroll-driven 3D storytelling, filtered through the codex: spectacle must be codex-legal (light, grain, displacement, parallax depth — no glow/gradients/glitch), typography timing frame-precise, WCAG AA / prefers-reduced-motion mandatory.

### Rules

1. **PROVISIONAL (high confidence, first-party)** — Wire Lenis to ScrollTrigger with the canonical 4-line sync and nothing else: `const lenis = new Lenis(); lenis.on('scroll', ScrollTrigger.update); gsap.ticker.add((t) => lenis.raf(t * 1000)); gsap.ticker.lagSmoothing(0);` plus `import 'lenis/dist/lenis.css'`. Keep Lenis defaults (lerp 0.1, duration 1.2) as the starting point; get the codex "slow orbit" feel from timeline easing, not from cranking Lenis smoothing. `lagSmoothing(0)` prevents GSAP from delaying scroll updates after a lag spike, which would desync 3D motion from text position. Source: https://github.com/darkroomengineering/lenis/blob/main/README.md

2. **VALIDATED (high)** — Structure each 3D "act" as ONE master GSAP timeline scrubbed by ONE ScrollTrigger over a tall scroll container: `gsap.timeline({ scrollTrigger: { trigger, start: 'top top', end: 'bottom bottom', scrub: 1 } })`. Use ~500vh for a single-scene act, ~900vh for a multi-shot act; `scrub: 0.5` for text (tighter sync), `scrub: 1` for camera/mesh moves. Never attach a ScrollTrigger to a tween that lives inside a timeline — parent timeline and scroll fight for the child's playhead. Container height IS the pacing control. Sources: https://tympanus.net/codrops/2025/11/19/how-to-build-cinematic-3d-scroll-experiences-with-gsap/ · https://gsap.com/resources/st-mistakes/ · https://svilenkovic.com/3d/how-to-make-scroll-driven-3d

3. **PROVISIONAL (high confidence)** — Never tween Three.js objects directly. Have GSAP animate plain proxy objects (`const cameraAnim = {x:0,y:0,z:8}` plus a separate lookAt target) and apply them every frame in the render loop via `camera.position.set(...)` + `camera.lookAt(...)`. Place each camera "shot" at a fractional timeline position (`tl.to(cameraAnim, {...}, shotStartPercent / 100)`) so shots map to exact scroll percentages and caption tweens placed at the same position land frame-precisely. Source: https://tympanus.net/codrops/2025/11/19/how-to-build-cinematic-3d-scroll-experiences-with-gsap/

4. **VALIDATED (high; two independent Codrops-published studios)** — Register a small family of named CustomEases at init and use only those: `cinematicSilk (0.45,0.05,0.55,0.95)`, `cinematicSmooth (0.25,0.1,0.25,1)`, `cinematicFlow (0.33,0,0.2,1)`, `cinematicLinear (0.4,0,0.6,1)`. For non-scrubbed section transitions: exits `power4.in` at ~0.8s, entrances `expo.out`, camera moves `expo.inOut` over ~2s, text lines staggered 0.1–0.2s. A locked ease vocabulary is the motion equivalent of locked color tokens — it makes `motion.orbit.slow` enforceable in code review. Sources: https://tympanus.net/codrops/2025/11/19/how-to-build-cinematic-3d-scroll-experiences-with-gsap/ · https://tympanus.net/codrops/2026/03/18/building-seamless-3d-transitions-with-webflow-gsap-and-three-js/

5. **VALIDATED (high; two independent Codrops tutorials converge)** — Drive shader effects from scroll VELOCITY, smoothed in the render loop — not from GSAP timelines. Pass a `uScrollVelocity` uniform (from Lenis); vertex displacement `position.y -= sin(uv.x * PI) * min(abs(uScrollVelocity), 5.0) * sign(uScrollVelocity) * 0.01`; simplex-noise grain at ~0.01 opacity factor mixed by `uScrollVelocity * 0.1`; smooth all uniforms with exponential lerp 0.05–0.15 per frame and momentum decay ~0.92. Velocity-driven effects feel physical (tectonic, per FORP) and self-settle to stillness — exactly the codex-legal spectacle palette: displacement + grain + light response, zero glow. Sources: https://tympanus.net/codrops/2024/07/18/how-to-create-distortion-and-grain-effects-on-scroll-with-shaders-in-three-js/ · https://tympanus.net/codrops/2025/11/19/how-to-build-cinematic-3d-scroll-experiences-with-gsap/

6. **PROVISIONAL (high confidence, first-party GSAP docs)** — Apply GSAP's pinning discipline verbatim: create pinned ScrollTriggers in document/scroll order (use `refreshPriority` for exceptions); use function-based end values (`end: () => '+=' + elem.offsetHeight`) plus `invalidateOnRefresh: true`; for multiple tweens on the same element use `.fromTo()` or `immediateRender: false`; call `ScrollTrigger.refresh()` after any async content load; force `html { scroll-behavior: auto !important; }`; slow a scrubbed animation by extending `end` distance (e.g. `'+=600'` vs `'+=300'`), never tween duration; kill triggers on SPA unmount. Source: https://gsap.com/resources/st-mistakes/

7. **VALIDATED (high)** — Make the WebGL canvas a persistent singleton that lives OUTSIDE the router/DOM-swap container, and clone already-loaded GLTF scenes instead of reloading them. Run the camera move between "rooms" in parallel with DOM exit/enter tweens so everything reads as one take. (r3f-scroll-rig: "GlobalCanvas should be kept outside of your router to prevent unmounting.") Sources: https://tympanus.net/codrops/2026/03/18/building-seamless-3d-transitions-with-webflow-gsap-and-three-js/ · https://github.com/14islands/r3f-scroll-rig

8. **PROVISIONAL (high confidence; single shipped-site postmortem)** — Support two scroll modes and switch dynamically per scene: free continuous scrub for atmospheric/parallax sections, and snap-block "beat" scrolling (GSAP Observer normalizing wheel/touch/trackpad into locked moments) for narrative key scenes. Beats guarantee dwell time (3-second hook rule, caption timing); atmosphere needs freedom; Observer unifies input devices. Source: https://tympanus.net/codrops/2026/04/28/more-than-a-portfolio-building-a-scroll-driven-3d-world-with-something-to-say/

9. **VALIDATED (medium)** — In React, pick exactly ONE scroll owner per page: either drei ScrollControls (`pages` = number of 100vh pages, `damping` 0.2–0.25, `useScroll().range/curve/visible`) OR Lenis+ScrollTrigger driving R3F refs. Do NOT mix ScrollTrigger `pin` with drei ScrollControls — they fight over the scroll container. If DOM text must track WebGL meshes 1:1, use 14islands' r3f-scroll-rig (requires predictable element heights — always set CSS aspect-ratio on images). Sources: https://drei.docs.pmnd.rs/controls/scroll-controls · https://github.com/14islands/r3f-scroll-rig · https://gsap.com/community/forums/topic/40114-scrolltrigger-pin-and-dreis-scrollcontrols-dont-play-well-together/

10. **VALIDATED (high)** — Enforce a hard performance budget uniformly: cap DPR at `Math.min(devicePixelRatio, 2)`; Draco-compress every GLB; use KTX2/Basis textures; GPU-instance every repeated element (measured: 30 FPS vs 144 from instancing alone); frustum-cull aggressively instead of static LODs; render atmospheric/particle passes at reduced resolution on mobile and composite back; use `frameloop="demand"` (R3F) or pause the render loop when nothing 3D is on screen. Dropped frames destroy "slow orbit" elegance faster than any wrong color. Sources: https://tympanus.net/codrops/2026/04/28/more-than-a-portfolio-building-a-scroll-driven-3d-world-with-something-to-say/ · https://tympanus.net/codrops/2026/03/18/building-seamless-3d-transitions-with-webflow-gsap-and-three-js/ · https://tympanus.net/codrops/2025/11/19/how-to-build-cinematic-3d-scroll-experiences-with-gsap/

11. **VALIDATED (high; three independent sources converge)** — Ship a three-tier degradation ladder, decided before first render: (1) full experience; (2) device-tier-detected light version (fewer particles, simpler shaders, reduced/no smooth scroll on weak mobile); (3) static fallback for no-WebGL and `prefers-reduced-motion: reduce` (matchMedia check, early return from animation callbacks, same narrative as composed stills + instant text). Use ResizeObserver, not window resize listeners. Sources: https://svilenkovic.com/3d/how-to-make-scroll-driven-3d · https://github.com/14islands/r3f-scroll-rig · https://tympanus.net/codrops/2026/03/18/building-seamless-3d-transitions-with-webflow-gsap-and-three-js/

12. **VALIDATED (high; two independent Codrops-published studios)** — Typography choreography numbers: SplitText character staggers 0.02s (0.01s for emphasis) with scrub 0.5 for scroll-scrubbed text; line staggers 0.1–0.2s for discrete transitions; place every text tween at the same fractional timeline position as its paired camera shot so they cannot drift. Document caption in/out points as timeline percentages per section (a caption cue sheet), not ad-hoc trigger offsets — positions are relative and survive refactors/resizes; pixel offsets are not. Sources: https://tympanus.net/codrops/2025/11/19/how-to-build-cinematic-3d-scroll-experiences-with-gsap/ · https://tympanus.net/codrops/2026/03/18/building-seamless-3d-transitions-with-webflow-gsap-and-three-js/

13. **PROVISIONAL (medium; article existence verified, implementation details not fetched)** — For long camera flythroughs, author the path as a curve in a real tool (Blender-exported paths, or Theatre.js sequences scrubbed by scroll in R3F) rather than hand-tuning `tl.to()` chains. Reserve hand-placed proxy-object shots for scenes with fewer than ~5 camera positions — DCC-authored CatmullRom/NURBS paths keep velocity continuous, which the codex ban on fast/jerky motion requires. Sources: https://tympanus.net/codrops/2023/02/14/animate-a-camera-fly-through-on-scroll-using-theatre-js-and-react-three-fiber/ · https://tympanus.net/codrops/2026/02/02/building-a-scroll-revealed-webgl-gallery-with-gsap-three-js-astro-and-barba-js/

## Lens 2 — Scroll-Scrubbed Cinematic Sequences (canvas frames vs `<video>` currentTime vs WebCodecs)

Motion-engineering for scrub techniques, encoding rules, payload budgets, alpha video, and the GLB-vs-pre-rendered decision — with Higgsfield kling3_0/seedance/GLB as the asset source.

### Rules

1. **VALIDATED (high; three independent sources)** — Make canvas-rendered frames (image sequence or WebCodecs-decoded video) the DEFAULT scrub technique for hero scenes; treat raw `<video>.currentTime` scrubbing as a fallback, never the primary path. Measured: naive currentTime scrubbing drops frames badly on mobile; pre-computed frames were 3–4.5x faster to interactive (244 frames, ~3MB over HTTP/2, 2.5s first load / 1.3s cached). Sources: https://www.ghosh.dev/posts/playing-with-video-scrubbing-animations-on-the-web/ · https://blog.yoanngueny.com/the-secrets-for-an-optimized-scroll-based-html5-video/ · https://github.com/dkaoster/scrolly-video

2. **VALIDATED (high; four independent sources)** — When you DO scrub `<video>.currentTime`, re-encode with dense keyframes: interval 1 (`-g 1`) for guaranteed smoothness, or every 5 frames for Chrome/Edge/Safari and every 2 for Firefox. Reference: `ffmpeg -i in.mp4 -vf scale=1280:-1 -movflags faststart -vcodec libx264 -crf 18 -g 2 -pix_fmt yuv420p -an out.mp4`. Ship BOTH H.264 MP4 and VP9 WebM. Budget for the ~5–6x file-size cost of dense keyframes (measured: MP4 146KB→845KB, WebM 195KB→1,038KB). Preload the whole file via fetch→blob→objectURL so scrubbing never waits on range requests. Sources: https://medium.com/@chrislhow/scroll-to-scrub-videos-4664c29b4404 · https://muffinman.io/blog/scrubbing-videos-using-javascript/ · https://blog.yoanngueny.com/the-secrets-for-an-optimized-scroll-based-html5-video/ · https://github.com/dkaoster/scrolly-video

3. **VALIDATED (high; three independent tutorials)** — Image-sequence budgets: 60–150 frames per pinned scene, WebP at ~80% quality, ≤1920x1080 source; expect ~50–80KB/frame, ~3–5MB per 60-frame sequence (vs 15–20MB PNG). Extract from Higgsfield video renders with `ffmpeg -i input.mp4 -vframes 60 frame-%03d.webp`. Since the codex mandates slow orbital motion, 60–90 frames over a long pin distance reads smooth — more frames only pay off for fast motion, which the codex forbids anyway. Sources: https://css-tricks.com/lets-make-one-of-those-fancy-scrolling-animations-used-on-apple-product-pages/ · https://www.in-sync.io/blog/tutorial-apple-scroll · https://gsapvault.com/blog/scroll-image-sequence-tutorial

4. **VALIDATED (high)** — Preloading discipline: download all frames of a sequence in parallel (promise-based) before the pin activates; keep them as decoded ImageBitmap objects, not `<img>` DOM nodes; request individual files over HTTP/2 rather than sprite sheets; defer initialization until the section approaches the viewport; provide a loading state and a low-bandwidth tier (Apple: 55.8MB desktop degrades to ~347KB on 3G — network tiering is part of the pattern, not optional). Sources: https://www.ghosh.dev/posts/playing-with-video-scrubbing-animations-on-the-web/ · https://gsapvault.com/blog/scroll-image-sequence-tutorial · https://css-tricks.com/lets-make-one-of-those-fancy-scrolling-animations-used-on-apple-product-pages/

5. **VALIDATED (medium; two independent sources + official docs)** — For the flagship scene, use the WebCodecs pipeline (video file → demux with mediabunny → VideoDecoder → VideoFrame buffer → canvas) with strict memory discipline: sliding buffer ~1s each direction (200ms forward / 300ms backward eviction), explicit `frame.close()` on evicted frames, skip intermediate seeks when >3 are queued, rAF-debounced rendering. Always pair with a currentTime fallback (keyframe-dense encode) for mobile Safari — WebCodecs is production-grade only on Chromium. Sources: https://lionkeng.medium.com/a-tutorial-webcodecs-video-scroll-synchronization-8b251e1a1708 · https://github.com/dkaoster/scrolly-video · https://scrollyvideo.js.org/

6. **VALIDATED (high; four independent implementations)** — Scroll-to-frame choreography constants: pin the canvas with ~400–500vh of scroll runway per sequence (or `end: '+=3000'` px); map `frameIndex = Math.min(frameCount-1, Math.floor(scrollFraction * frameCount))`; use `scrub: 0.5` with `snap: 'index'` to land on whole frames; render inside requestAnimationFrame; scale the canvas by devicePixelRatio. For real-time 3D, ease toward scroll targets with lerp factor ≈ `min(dt*6, 1)`. Text beats become ScrollTrigger positions on the same timeline as frame indices — this is how "perfect timing between animations and text locations" is achieved. Sources: https://css-tricks.com/lets-make-one-of-those-fancy-scrolling-animations-used-on-apple-product-pages/ · https://gsapvault.com/blog/scroll-image-sequence-tutorial · https://www.in-sync.io/blog/tutorial-apple-scroll · https://www.builder.io/blog/webgl-scroll-animation

7. **VALIDATED (high)** — Alpha-channel video (smoke/vapor, golden-particle layers over Deep Indigo): dual-encode — HEVC-with-alpha .mov listed FIRST in `<video>` (`type='video/quicktime; codecs=hvc1...'`) for Safari, plus VP9 `yuva420p` WebM for Chrome/Firefox. Better 2024+ path: STACKED-ALPHA — vstack color + grayscale alpha into one normal video (AV1/HEVC), recombine in a WebGL shader (npm: `stacked-alpha-video`); measured 460KB AV1 / 1.14MB HEVC fallback vs 1.1MB VP9-alpha / 3.4MB HEVC-alpha, with ~8KB alpha overhead. Encode 10-bit (`yuva444p10le`) to avoid limited-range banding on soft vapor edges. 720p is often big enough for short web clips. Sources: https://jakearchibald.com/2024/video-with-transparency/ · https://rotato.app/blog/transparent-videos-for-the-web

8. **VALIDATED (medium; three sources argue the tradeoff)** — GLB vs pre-rendered decision rule: use real-time Three.js/R3F with Higgsfield GLBs only when the element must respond continuously (pointer parallax, morph targets across the whole scroll range, resolution-independence full-bleed). Use pre-rendered Higgsfield video/frame sequences when the look is cinematic/photoreal lighting and the only interaction is scrubbing. Hybrid is the sweet spot: GLB for the one hero object (hummingbird), pre-rendered sequences for atmosphere. For GLBs: compress textures, decimate meshes, device-tier detect, static-image fallback without WebGL, glTF 2.0 as archival format. Sources: https://www.builder.io/blog/webgl-scroll-animation · https://svilenkovic.com/3d/scrollytelling-trends-2026 · https://www.ghosh.dev/posts/playing-with-video-scrubbing-animations-on-the-web/

9. **VALIDATED (high)** — iOS hard rules: every `<video>` gets `playsinline` + `muted`; assume Low Power Mode kills video entirely (no workaround) — so the mobile fallback chain must terminate in IMAGES (WebP sequence or static poster), never in another video. Test scrub feel on real devices; Safari scrubs currentTime well but older iPhones struggle. Sources: https://github.com/dkaoster/scrolly-video · https://muffinman.io/blog/scrubbing-videos-using-javascript/ · https://jakearchibald.com/2024/video-with-transparency/

10. **VALIDATED (high)** — Under `prefers-reduced-motion: reduce`, skip pinning and scrubbing entirely and render each sequence's first (or most legible) frame as a static image with the same typography beats laid out linearly. Page budget: stay near 3–5MB-per-scene WebP discipline, never Apple's 55.8MB desktop payload; adaptive-serve a few-hundred-KB tier on slow connections. Sources: https://gsapvault.com/blog/scroll-image-sequence-tutorial · https://css-tricks.com/lets-make-one-of-those-fancy-scrolling-animations-used-on-apple-product-pages/

## Lens 3 — Typography Choreography & the Accessibility Floor

Syncing text reveals to scroll progress, pin-vs-flow decisions, readability over motion, WCAG 2.3.3 / prefers-reduced-motion — filtered through codex constraints (motion.orbit.slow, no glow/gradients, WCAG AA locked).

### Rules

1. **VALIDATED (high; vendor docs + standards reference)** — Give every section an explicit, named scrub window: in GSAP, two-part start/end mapping (`start: 'top top', end: '+=2000'`) with a timeline per section; in native CSS, `view-timeline` + `animation-range` (e.g. `entry 0% cover 100%`) with a required `@supports` fallback. One timeline per section, never one global page timeline. The scrollbar becomes the timeline controller and each text beat gets a fixed address in scroll space. Sources: https://gsap.com/docs/v3/Plugins/ScrollTrigger/ · https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_scroll-driven_animations

2. **PROVISIONAL (high confidence, first-party GSAP docs)** — Use numeric scrub smoothing, not instant scrub: `scrub: 0.5–1` rather than `scrub: true`. Numeric scrub is the only scrub mode that low-pass-filters user scroll velocity into slow motion (codex law: never fast). Pair with `fastScrollEnd: true` (fast scrollers get completed animations, not half-revealed text) and `anticipatePin: 1` (no flash of unpinned content on high-velocity scrolls). Source: https://gsap.com/docs/v3/Plugins/ScrollTrigger/

3. **VALIDATED (high)** — Adopt the sticky-graphic pattern as the default architecture: the 3D canvas is pinned (CSS `position: sticky` or GSAP `pin: true`) while text steps flow past in normal document flow (side-by-side or text-overlay layouts). Constraints: animate only nested children of the pinned element, never the pinned element itself; pinning breaks inside any transform-carrying ancestor; avoid `content-visibility: auto`. Sticky degrades gracefully — unsupporting browsers render the element static in source order. Sources: https://pudding.cool/process/scrollytelling-sticky/ · https://gsap.com/docs/v3/Plugins/ScrollTrigger/ · https://pudding.cool/process/introducing-scrollama/

4. **PROVISIONAL (high confidence; single research org — the only source with observed user behavior)** — Keep critical copy OUT of scrub-timed reveals. NN/g user testing: pages combining altered scroll rates with readable text "created the most severe usability failures"; users cannot retain appearing-and-disappearing text in working memory. Rule: headlines and decorative display type (Cormorant one-thought-per-page H1s) may scrub; body copy and captions (Montserrat) must reach a full-opacity resting state and hold it. Practical beat: transition in over ~first 15% of the step's range, hold fully legible ~70%, transition out over last ~15% (split is a synthesis; the dominant legible dwell is NN/g's finding). Source: https://www.nngroup.com/articles/scrolljacking-101/

5. **VALIDATED (high; research + practitioner convergence)** — Alternate choreographed sections with normal-scroll sections; keep scroll choreography below the fold; never convert vertical scroll gestures into horizontal movement. Reserve pinned scroll-scrub sequences for content where the transition itself carries meaning (change over time, spatial movement). Sources: https://www.nngroup.com/articles/scrolljacking-101/ · https://pudding.cool/process/responsive-scrollytelling/

6. **PROVISIONAL (high confidence, first-party GSAP docs)** — Snap scroll position to timeline labels so text always resolves to a legible resting beat: `snap: 'labels'` (label each text beat) or explicit progress values (`snap: [0, 0.5, 1]`); tune with `duration: {min: 0.2, max: 3}` clamped by velocity, `delay: 0.2`, `directional: true`, gentle ease like `power1.inOut`. Guarantees a stopped scroll never leaves a caption at 40% opacity. Source: https://gsap.com/docs/v3/Plugins/ScrollTrigger/

7. **VALIDATED (high; four independent accessibility sources)** — Enforce contrast at the WORST frame of the motion behind text: body text 4.5:1, large text 3:1, validated at multiple points of the animation including the lowest-contrast area of the busiest frame. Techniques ranked for this codex: (1) solid text container — Deep Indigo #171D2D panel or Pearl Ivory #F4F1ED card, 28px radius, hairline stroke; (2) uniform semi-transparent SOLID scrim between motion layer and text; (3) locally blurring or darkening the region behind text. Gradient scrims are codex-forbidden — scrims must be flat single-color layers. Sources: https://www.w3.org/TR/WCAG20-TECHS/G18.html · https://webaim.org/articles/contrast/ · https://webflow.com/accessibility/checklist/task/check-the-contrast-for-text-that-overlaps-images-or-videos · https://www.wcag.com/blog/content-over-images-how-does-this-ux-ui-trend-impact-accessibility/

8. **VALIDATED (high; W3C + Google)** — Treat prefers-reduced-motion as a hard requirement implemented in BOTH layers: CSS (`@media (prefers-reduced-motion: reduce)` disabling transform/parallax animations; optionally gate the animation stylesheet via `<link media="(prefers-reduced-motion: no-preference)">`) AND JS (`window.matchMedia('(prefers-reduced-motion: reduce)')` with a change listener to kill/rebuild ScrollTriggers live). Replacement grammar: opacity cross-fades instead of movement (SC 2.3.3 excludes opacity/color from "motion"), static poster frames via `<picture media>` instead of scrub video, no parallax, no pinning; the full narrative text remains readable in document flow. Scrolling is an interaction under WCAG SC 2.3.3, and parallax-on-scroll is W3C's canonical non-essential case. Sources: https://www.w3.org/WAI/WCAG21/Understanding/animation-from-interactions · https://www.w3.org/WAI/WCAG22/Techniques/css/C39 · https://web.dev/articles/prefers-reduced-motion

9. **VALIDATED (high; practitioner + research convergence)** — Mobile discipline: never size step containers or triggers in vh units (mobile browser chrome show/hide changes viewport height and desynchronizes every trigger) — compute px heights from `window.innerHeight` on load/resize. Fork layout with matchMedia (Pudding example breakpoint: 800px). On small screens either radically shorten the scrolly or stack it into static text + stills; NN/g: scrolljacking benefits "rarely justify implementation costs" on mobile. Designing the mobile-stacked version first also forces identification of essential text beats (codex negative-space pressure). Sources: https://pudding.cool/process/responsive-scrollytelling/ · https://www.nngroup.com/articles/scrolljacking-101/

10. **VALIDATED (medium; vendor docs + practitioner postmortems)** — Text animation performance rules: animate only transform and opacity on type (never layout properties like top/width/font-size); character/word-split reveals read as premium and do not require pinning — reserve them for display type; use `ScrollTrigger.batch()` for many small text triggers; use `will-change` sparingly; kill triggers on section teardown. Keep pinning for the 3D canvas and let type flow (mobile pin fragility: dynamic viewport + font-loading shifts break pin calculations). Sources: https://gsap.com/docs/v3/Plugins/ScrollTrigger/ · https://dev.to/xuanhai0913/gsap-scrolltrigger-pin-true-nearly-broke-my-portfolio-heres-what-i-learned-28i7 · https://dev.to/vishwark/mastering-gsap-scrolltrigger-a-complete-practical-guide-5bi3

## Lens 4 — Web-Ready 3D/Motion Asset Production (Higgsfield pipeline + GLB optimization + loops + alpha)

What Higgsfield verifiably provides (checked against live MCP tool schemas, corroborated by higgsfield.ai) plus independently-sourced engineering practices — under a 6,000-credit budget where cost preflight is a first-class practice.

### Rules

1. **PROVISIONAL (high confidence; first-party MCP schema is the only spec)** — Higgsfield 3D pipeline: use model `image_to_3d` for general image-to-GLB (optional texturing, PBR, rigging); `multi_image_to_3d` with 2–4 views of the same subject for better geometric accuracy; `sam_3_3d` to lift a single object out of a busy still (only 3D model that accepts a prompt); `3d_rigging` to rig an existing GLB (takes a prior 3D job_id or https GLB URL via `model_url`, not images). For animated rigs, search the `animation_actions` catalog (678 actions: WalkAndRun / BodyMovements / DailyActions / Dancing / Fighting, each with preview GIF) and pass `animation_action_id` with `enable_animation:true`. The marketing site does NOT document GLB/PBR/rigging specs — the schema is the spec. Sources: https://higgsfield.ai/ · https://higgsfield.ai/apps/3d-render

2. **PROVISIONAL (high confidence; stated in the generate_3d schema)** — Art-direct in 2D before going to 3D: the mesh reproduces only what is in the source image. To add/change props, clothing, or held objects, edit the image first with generate_image (nano_banana_pro for 4K stills/text; soul_2 for portraits), then convert. This is also the cheapest iteration loop — lock composition/materials at the still stage. Source: https://higgsfield.ai/

3. **PROVISIONAL (high confidence; schema-verified)** — Preflight every Higgsfield job with `get_cost:true` — supported on generate_3d, generate_image, generate_video, and upscale_image (flat cost); NOT supported on upscale_video. ## Merged Pitfall Register

Deduplicated across all five lenses; entries flagged by 2+ lenses are marked **[multi-lens]**.

### GSAP / ScrollTrigger

1. Nesting a ScrollTrigger inside a tween that lives in a timeline — parent timeline and scroll fight for the child's playhead; animation stutters or ignores scroll (https://gsap.com/resources/st-mistakes/).
2. Multiple `.to()` tweens on the same element/property across sections — cached start values cause visible snap-backs; fix with `.fromTo()`, `immediateRender: false`, or one timeline (https://gsap.com/resources/st-mistakes/).
3. CSS `scroll-behavior: smooth` anywhere on the page silently corrupts `ScrollTrigger.refresh()` position math — force `html { scroll-behavior: auto !important; }` (https://gsap.com/resources/st-mistakes/).
4. Creating pinned ScrollTriggers out of scroll/document order without `refreshPriority` — later pins compute wrong start/end and every section after the first pin drifts (https://gsap.com/resources/st-mistakes/).
5. Hard-coded start/end pixel values that don't recalc on resize/orientation change — use function-based values + `invalidateOnRefresh: true`, and `ScrollTrigger.refresh()` after lazy-loaded content changes document height (https://gsap.com/resources/st-mistakes/; https://github.com/14islands/r3f-scroll-rig mandates predictable heights via CSS aspect-ratio).
6. Trying to slow a scrubbed animation by increasing tween duration — durations inside a scrubbed timeline are only proportional; extend the scroll distance instead (`end: '+=600'`) (https://gsap.com/resources/st-mistakes/).
7. Scrubbed tweens starting at scroll position 0 visually "jump" on page load — offset the start or use ScrollTrigger's `clamp()` (https://gsap.com/resources/st-mistakes/).
8. Animating the pinned element itself, pinning inside a transform-carrying ancestor (common when the WebGL canvas wrapper has a transform), or using `content-visibility: auto` — all documented ScrollTrigger breakage modes that shift text/animation alignment by hundreds of px (https://gsap.com/docs/v3/Plugins/ScrollTrigger/).
9. No resting state: without snap-to-labels, a user who stops scrolling mid-window is left with 40%-opacity captions and half-translated headlines — the page's most common visual state becomes an undesigned one (https://gsap.com/docs/v3/Plugins/ScrollTrigger/).

### Scroll ownership / Lenis

10. **[multi-lens]** Mixing scroll owners: ScrollTrigger `pin` + drei ScrollControls conflict; Lenis + drei ScrollControls is the same double-ownership trap. Pick exactly one scroll authority per page (https://gsap.com/community/forums/topic/40114-scrolltrigger-pin-and-dreis-scrollcontrols-dont-play-well-together/).
11. Lenis hard limitations to design around, not discover in QA: no CSS scroll-snap (use lenis/snap), smooth scroll dies over iframes, `position: fixed` lags on pre-M1 macOS Safari, capped to 60fps on Safari, syncTouch unstable on iOS<16 (https://github.com/darkroomengineering/lenis/blob/main/README.md).
12. **[multi-lens]** Shipping desktop smooth-scroll settings to mobile — smooth scrollbar + scrolling WebGL is "usually laggy" on mobile; degrade to native scroll or a much lower smoothing tier (https://github.com/14islands/r3f-scroll-rig).

### Video scrubbing / encoding

13. **[multi-lens]** Scrubbing `<video>.currentTime` with default keyframe intervals (Adobe Media Encoder default: 72) — every seek walks back to the previous keyframe; the classic jaggy scrub, and the single most common failure of the technique (https://blog.yoanngueny.com/the-secrets-for-an-optimized-scroll-based-html5-video/; https://medium.com/@chrislhow/scroll-to-scrub-videos-4664c29b4404; https://css-tricks.com/lets-make-one-of-those-fancy-scrolling-animations-used-on-apple-product-pages/).
14. Testing only in Chrome: Firefox scrubs H.264 MP4 choppily regardless of keyframe density (needs VP9 WebM, keyframes every ~2 frames); Safari is fine with sparse keyframes — a Chrome-only test hides both facts (https://muffinman.io/blog/scrubbing-videos-using-javascript/).
15. Forgetting keyframe-dense encodes cost ~5–6x file size (measured: MP4 146KB→845KB, WebM 195KB→1,038KB from g=100 to g=5) (https://muffinman.io/blog/scrubbing-videos-using-javascript/).
16. **[multi-lens]** iOS Low Power Mode silently disables ALL video with no workaround — any fallback chain ending in `<video>` renders a dead black hero for a large slice of mobile users; the chain must terminate in images (https://github.com/dkaoster/scrolly-video).
17. Scrubbing directly in the scroll event handler on Android/native scroll — jank is structural; decouple with a lerped rAF loop (or virtual scroll) and snap to whole frame indices (https://blog.yoanngueny.com/the-secrets-for-an-optimized-scroll-based-html5-video/).
18. WebCodecs VideoFrame objects hold GPU memory the garbage collector never frees — without explicit `frame.close()` and a bounded sliding buffer the page leaks until the tab dies; WebCodecs is production-grade only on Chromium, so a currentTime fallback stays mandatory (https://lionkeng.medium.com/a-tutorial-webcodecs-video-scroll-synchronization-8b251e1a1708; https://github.com/dkaoster/scrolly-video).
19. Sprite sheets for frame sequences are counterproductive under HTTP/2 — individual frame files measured faster (https://www.ghosh.dev/posts/playing-with-video-scrubbing-animations-on-the-web/).
20. **[multi-lens]** Copying Apple's desktop payload (55.8MB / 1,609 requests / 148 frames) without their adaptive tier (~347KB on 3G) — the pattern only ships responsibly with device/network tiering and a preload UI (https://css-tricks.com/lets-make-one-of-those-fancy-scrolling-animations-used-on-apple-product-pages/).

### Alpha video

21. **[multi-lens]** Safari cannot play VP9-with-alpha (renders opaque black, no error) and picks the wrong `<source>` unless the HEVC .mov (`codecs=hvc1`) is listed FIRST; Chrome Android has had alpha-rendering bugs; animated AVIF alpha (504KB) cannot hold 60fps even on high-end laptops; ffmpeg's hevc_videotoolbox output is ~2x Apple Compressor's size (https://jakearchibald.com/2024/video-with-transparency/; http://terhech.de/posts/2025-02-02-transparent-video-safari.html).
22. Limited-range 8-bit YUV (16–235) causes visible banding on soft alpha edges (smoke, vapor) — encode 10-bit `yuva444p10le` and premultiply alpha for the HEVC path (https://jakearchibald.com/2024/video-with-transparency/).
23. remove_background(`media_type:'video'`) is schema-verified, but the schema does not state the output container or whether alpha survives — verify the first output and plan to re-encode to VP9-alpha + HEVC-alpha before committing credits to a batch (https://higgsfield.ai/).

### Asset pipeline / Higgsfield / GLB

24. Higgsfield's marketing pages do not document the GLB pipeline (/apps/3d-render is a 2D photo effect; /apps/3d 404s); output polygon count and texture resolution are documented nowhere — inspect and optimize every mesh, never assume it fits budget as-generated (https://higgsfield.ai/apps/3d-render).
25. Draco cannot compress morph targets or keyframe animation — compressing a rigged Higgsfield GLB with Draco leaves animation data untouched; use meshopt for rigged assets. For geometry under ~1MB the Draco WASM decoder can cost more than it saves (https://github.com/donmccurdy/glTF-Transform/discussions/347; https://discourse.threejs.org/t/gltf-and-or-glb-online-optimization-and-compression/62753).
26. Credit traps (6,000-credit budget): `get_cost` preflight exists on generate_3d/image/video and upscale_image but NOT upscale_video; bytedance video-upscale fps above 30 doubles cost; upscale_image requires exact source width/height — the server does not infer them (https://higgsfield.ai/).
27. The "same still as first and last frame" loop prompt usually produces near-static motion; use the two-clip bridge method — and lossless concat (`-c copy`) silently fails or corrupts unless both clips share identical codec, resolution, and pixel format (https://ffmpeg.party/guides/ai-video-loop/).
28. Khronos 2.0 dropped fixed numeric budgets (defers to Asset Auditor profiles) — cite the 1.0 Publishing Targets table for hard numbers, and treat a multi-asset scroll page as sharing ONE budget, not one per mesh (https://raw.githubusercontent.com/KhronosGroup/3DC-Asset-Creation/main/asset-creation-guidelines-1.0/full-version/sec99_PublishingTargets/PublishingTargets.md).
29. **[multi-lens]** Frame-sequence size explosion when done naively: 90 PNG frames can outweigh an equivalent 3s video ~29:1 (56MB vs 1.92MB) — cap scenes at 60–150 WebP q80 frames, preload per section, poster fallback (https://css-tricks.com/lets-make-one-of-those-fancy-scrolling-animations-used-on-apple-product-pages/).
30. Uncapped devicePixelRatio and non-instanced repeated meshes: instancing alone measured 30 FPS vs 144; cap DPR at 2 and instance all repeated geometry (https://tympanus.net/codrops/2026/04/28/more-than-a-portfolio-building-a-scroll-driven-3d-world-with-something-to-say/).
31. **[multi-lens]** Shipping 4K textures/frames to a 6-inch screen quadruples payload for no visible gain — 4K is the master/archive tier; delivery is 1024–2048px textures and ~1600–1920px frames (https://raw.githubusercontent.com/KhronosGroup/3DC-Asset-Creation/main/asset-creation-guidelines-1.0/full-version/sec99_PublishingTargets/PublishingTargets.md).

### Typography / UX / accessibility

32. Sizing steps or triggers in vh on mobile — browser URL-bar show/hide changes viewport height and silently desynchronizes every text beat from its animation; compute px from `window.innerHeight` on load/resize (https://pudding.cool/process/responsive-scrollytelling/).
33. Scrubbing body copy — NN/g found text + altered scroll pace is the single worst scrolljacking failure; users cannot hold appearing/disappearing text in working memory, and long scrolljacks get mistaken for bugs (users refresh or leave). Scrub display type; rest body type (https://www.nngroup.com/articles/scrolljacking-101/).
34. Measuring text contrast on one flattering frame — with motion behind type, contrast is time-varying; the lowest-contrast frame/region must pass 4.5:1 (body) / 3:1 (large); automate a worst-frame check in QA (https://www.w3.org/TR/WCAG20-TECHS/G18.html; https://webflow.com/accessibility/checklist/task/check-the-contrast-for-text-that-overlaps-images-or-videos).
35. Misreading WCAG 2.3.3's scrolling exemption — parallax layers, morphing 3D, and scrub-linked transforms are non-essential animation added to the scroll interaction and MUST be disableable; CSS-only prefers-reduced-motion is insufficient if the GSAP/JS layer keeps animating — matchMedia with a change listener is required (https://www.w3.org/WAI/WCAG21/Understanding/animation-from-interactions; https://web.dev/articles/prefers-reduced-motion).
36. **[multi-lens; all four engineering lenses]** Omitting or deferring the reduced-motion tier — it is codex law (WCAG AA), it must not lose content (all copy readable statically; opacity cross-fades permitted since SC 2.3.3 excludes opacity/color from "motion"), and it affects ASSET PLANNING: composed stills of every key beat must be generated alongside GLB/video (plan Higgsfield 4K stills up front, not after) (https://www.w3.org/WAI/WCAG22/Techniques/css/C39; https://gsapvault.com/blog/scroll-image-sequence-tutorial).
37. Vertical-gesture-to-horizontal-motion mapping caused outright comprehension failures in NN/g testing — if a horizontal 3D dolly is wanted, keep the text track vertical and let only the camera move laterally (https://www.nngroup.com/articles/scrolljacking-101/).
38. Native CSS scroll-driven animations (`animation-timeline`/`animation-range`) shipped without an `@supports` guard — unsupported browsers get no animation AND no fallback state unless explicitly styled (https://developer.mozilla.org/en-US/docs/Web/CSS/CSS_scroll-driven_animations).
39. Gradient scrims as the default readability fix — standard industry advice, but forbidden by the codex; use flat single-color scrims, solid Deep Indigo/Pearl Ivory panels (28px radius, hairline stroke), or localized blur (https://www.wcag.com/blog/content-over-images-how-does-this-ux-ui-trend-impact-accessibility/).

### Codex governance / award-reference traps

40. **[multi-lens]** Interpreting "amazing VFX" as bloom/glow/neon post-processing — every effect in the verified sources that reads cinematic without glow is velocity-driven displacement, film grain, light/density modulation, and parallax depth (the codex-legal set). Any UnrealBloomPass-style pipeline is a governance violation, not a style choice (https://tympanus.net/codrops/2024/07/18/how-to-create-distortion-and-grain-effects-on-scroll-with-shaders-in-three-js/).
41. Copying award winners' compose stacks wholesale: Igloo Inc's SOTY kit includes shader text glitches/scrambles and chromatic aberration; Shader.se's compose pass includes bloom + chromatic aberration — glitch and bloom/glow are codex-banned. Strip to grain + fog + displacement + frost-style masking; replace text scramble with SDF-offset weight/tracking morphs (https://www.awwwards.com/igloo-inc-case-study.html; https://tympanus.net/codrops/2026/05/19/80s-business-tech-seamless-scene-transitions-inside-shader-ses-scroll-driven-webgpu-pipeline/).
42. Hubtown's signature is a GLOWING monolith — the reveal mechanic is stealable, the emissive glow is not; recreate presence with rim light, practical light sources, and fog occlusion (https://www.utsubo.com/blog/best-threejs-websites-2026).
43. WebGL-rendered UI text (Igloo renders its entire UI in-engine) conflicts with mandatory WCAG AA: canvas text has no DOM, no screen-reader path, uncontrolled contrast. Keep body/caption text in DOM with approved Ivory/Indigo pairings; reserve in-engine SDF text for display type only (https://www.awwwards.com/igloo-inc-case-study.html).
44. None of the fetched award case studies (Pendragon, Igloo, Cartier, Shader.se, Lusion) document prefers-reduced-motion fallbacks or fps targets — award-level does not mean accessible; the reduced-motion tier must be designed, not copied, and the only hard performance numbers available are Lusion's KB budgets and Shader.se's pass-skipping. Do not invent fps claims with citations.
45. The flashy pole also wins (Lando Norris took SOTY 2025 with lime-green kinetic typography) — juries reward execution quality at both poles, so "restrained" is a brand decision, not an award requirement. Argue restraint from the codex + quiet-luxury positioning, not "flashy sites lose" (https://metabole.studio/en/blog/immersive-website-examples).
46. Roundup listicles (Utsubo, Metabole) contain zero performance metrics and occasionally soft award attributions; secondary-summary-only descriptions (My Web Lab's "brutalist typography + cinematic dark transitions", Microsoft.ai's "vegetal shaders") must be verified against the live sites before any client-facing panel. Metabole's engagement stats (40% longer sessions, 93% more effective) are studio-reported marketing numbers — directional framing only, not audited facts (https://www.utsubo.com/blog/best-threejs-websites-2026; https://metabole.studio/en/blog/immersive-website-examples).
47. **[multi-lens]** Scroll-scrubbed spectacle without disposal discipline kills mid-range devices: Cartier disposes/loads scenes per alcove; Shader.se skips whole render passes off-screen ("no draw calls, no GPU work") — a single persistent scene graph across all universes would blow both the credit budget and the WCAG-mandated performance floor (https://www.webgpu.com/showcase/cartier-watches-and-wonders-immersive-garden/; https://tympanus.net/codrops/2026/05/19/80s-business-tech-seamless-scene-transitions-inside-shader-ses-scroll-driven-webgpu-pipeline/).
